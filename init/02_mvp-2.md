# 2차 MVP 개발 오케스트레이션 분해안

## 기준 기능

2차 MVP는 1차 MVP에서 만든 기본 커밋 흐름 위에 아래 기능을 추가하는 단계입니다.

1. --model 명령어 구현
2. AI Provider 구조 분리
3. localLLM 연동
4. localLLM endpoint 설정
5. /v1/models 기반 로컬 모델 목록 조회
6. 로컬 모델 선택 UI
7. API Key 인증 방식 구현
8. Gemini 또는 OpenAI-compatible Provider 1개 이상 연동
9. --push 구현
10. --reset 구현

## 0. 2차 MVP 핵심 목표

1차 MVP가 아래 흐름까지였다면,

convention

→ 변경 사항 확인

→ Mock 커밋 메시지 생성

→ confirm
→ git add
→ git commit

2차 MVP는 아래 흐름까지 확장하는 것이 목표입니다.
convention --model
→ AI Provider 선택
→ 인증 방식 선택
→ 모델 버전 선택
→ 설정 저장

convention
→ 실제 localLLM 또는 API Provider로 커밋 메시지 생성
→ confirm
→ git commit

convention --push
→ 커밋 후 git push

convention --reset
→ 최근 커밋 취소

## 1. 전체 개발 순서 요약

2차 MVP는 아래 순서로 진행하는 것이 좋습니다.

1. 1차 MVP 상태 점검
2. Provider 공통 인터페이스 정의
3. Mock Provider를 Provider 구조로 이전
4. localLLM Provider 구현
5. localLLM endpoint 설정 구현
6. /v1/models 모델 목록 조회 구현
7. 로컬 모델 선택 UI 구현
8. --model CLI 라우팅 구현
9. --model 대화형 설정 플로우 구현
10. API Key 입력/저장 구조 구현
11. API Key 기반 Provider 1개 구현
12. AI 호출 로직을 Provider 라우팅 방식으로 교체
13. --push 구현
14. --reset 구현
15. 보안 확인 및 통합 테스트

## 2. 2차 MVP용 Agent 목록

### A. 1차 MVP 점검 Agent

역할
2차 MVP 작업 전에 1차 MVP가 정상 동작하는지 확인합니다.

담당 작업

1. convention --help 확인
2. convention --set-mode 확인
3. convention --language 확인
4. convention --batch 확인
5. convention --step 확인
6. Mock AI 커밋 흐름 확인
7. confirmBeforeCommit 동작 확인

테스트 명령어
convention --help
convention --set-mode batch
convention --language ko
convention --batch
convention --step
convention

완료 기준
1차 MVP 기능이 깨지지 않고 정상 동작한다.

### B. Provider 인터페이스 정의 Agent

역할
AI Provider들이 공통으로 따라야 할 인터페이스를 정의합니다.
구현 대상 파일
src/providers/index.js
src/providers/mock.js
src/providers/types.js 또는 src/providers/contracts.js

담당 작업

1. Provider 공통 함수명 정의
2. generateCommitMessage 함수 인터페이스 정의
3. listModels 함수 인터페이스 정의
4. validateConfig 함수 인터페이스 정의
5. Provider 라우팅 구조 초안 작성

권장 인터페이스
generateCommitMessage({ prompt, config })
listModels(config)
validateConfig(config)

완료 기준
Provider가 바뀌어도 core/ai.js는 같은 방식으로 호출할 수 있다.

## 8-3. 커밋 확인 질문 설정 (`--question`, `-q`)

`convention --question` 또는 `convention -q`는 커밋 메시지 생성 후 사용자에게 커밋 여부를 물어볼지 설정하는 명령입니다.

- 설정 키는 기존 `confirmBeforeCommit`을 사용합니다.
- 기본값은 `true`이며, 생성된 커밋 메시지를 보여준 뒤 커밋 여부를 확인합니다.
- `false`로 설정하면 `convention`, `convention --step`, `convention --batch` 실행 시 커밋 메시지 생성 후 별도 질문 없이 staging 및 commit을 진행합니다.
- 설정 UI는 `prompts` select를 사용하며, 방향키로 `true` 또는 `false`를 선택하고 Enter로 저장합니다.
- `--question`은 설정 명령이므로 실행 후 commit flow를 시작하지 않습니다.
- `convention --help`와 README 사용법에 이 옵션을 표시합니다.
