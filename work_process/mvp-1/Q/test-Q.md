# Phase Q Git add Agent Test

`research-Q.md`의 구현 기준을 바탕으로, `addAll()` 및 `addFile(file)`이 1차 MVP 요구사항을 충족하는지 확인하기 위한 테스트 항목입니다.

## 1. 정적 검증 항목

| ID | 테스트 항목 | 검증 방법 | 예상 결과 |
| :--- | :--- | :--- | :--- |
| V-1 | 함수 export 확인 | `src/core/git.js` 코드 확인 | `addAll`, `addFile` 함수를 import할 수 있다. |
| V-2 | Git 명령 인자 확인 | Git command 실행 코드 확인 | `git add -A` 및 `git add -- <file>` 구조를 사용한다. |
| V-3 | 인자 배열 방식 확인 | Git command 실행 코드 확인 | 파일 경로를 shell 문자열이 아니라 argv 배열의 단일 인자로 전달한다. |
| V-4 | `--` pathspec 구분자 확인 | Git args 확인 | `addFile`에서 `["add", "--", file]` 구조를 사용한다. |
| V-5 | 에러 로깅 확인 | `try...catch` 및 Logger 사용 확인 | 명령 실패 시 Logger를 통해 에러를 기록한다. |
| V-6 | 예외 전파 확인 | 함수 구현 확인 | 치명적 실패 시 예외(throw)를 발생시켜 상위 흐름에 알린다. |

## 2. 기능 테스트 항목

### T-1: 전체 파일 스테이징 (Batch)
- **준비:** 여러 개의 파일을 수정 및 생성한다.
- **실행:** `addAll()`을 호출한다.
- **예상 결과:** `git status --porcelain` 확인 시 모든 변경 파일이 `A` 또는 `M` (staged) 상태로 표시된다.

### T-2: 개별 파일 스테이징 (Step)
- **준비:** 파일 A와 파일 B를 수정한다.
- **실행:** `addFile("a.js")`를 호출한다.
- **예상 결과:** `a.js`는 staged 상태가 되고, `b.js`는 여전히 unstaged 상태를 유지한다.

### T-3: 공백 포함 파일명 스테이징
- **준비:** `file with space.js` 파일을 생성한다.
- **실행:** `addFile("file with space.js")`를 호출한다.
- **예상 결과:** 파일명이 분리되지 않고 정확히 해당 파일만 staged 상태가 된다.

### T-4: 한글 파일명 스테이징
- **준비:** `한글파일.js` 파일을 생성한다.
- **실행:** `addFile("한글파일.js")`를 호출한다.
- **예상 결과:** 한글 경로가 깨지지 않고 정상적으로 staged 상태가 된다.

### T-5: 삭제된 파일 스테이징
- **준비:** 기존에 tracked 상태였던 `delete-me.js`를 삭제한다.
- **실행:** `addFile("delete-me.js")` 또는 `addAll()`을 호출한다.
- **예상 결과:** 삭제 내역이 staged 상태가 된다.

### T-6: 에러 처리 테스트
- **준비:** 존재하지 않는 파일 경로로 `addFile("none.js")`를 시도하거나, Git 저장소가 아닌 곳에서 실행한다.
- **실행:** 해당 함수를 호출한다.
- **예상 결과:** 예외가 발생하고 로거에 에러가 기록된다.

## 3. 테스트 절차

1. 원래 작업 디렉터리를 `originalCwd`로 저장한다.
2. 임시 디렉터리에 테스트용 Git 저장소를 생성한다.
3. `git init` 및 기본 사용자 정보를 설정한다.
4. 테스트용 파일들을 생성하고 초기 커밋을 진행하여 tracked 상태를 만든다.
5. 각 케이스별로 파일을 수정, 삭제, 생성한다.
6. `process.chdir(repoDir)` 상태에서 `addAll()` 또는 `addFile(file)`을 호출한다.
7. `execFileSync("git", ["status", "--porcelain"])` 등을 실행하여 실제 staged 여부를 검증한다.
8. 테스트 종료 후 `process.chdir(originalCwd)`로 복원한다.

## 4. 테스트 환경 주의사항

- **실제 데이터 보호:** 사용자 저장소의 실제 변경 사항을 의도치 않게 스테이징하지 않도록 주의합니다.
- **격리 검증:** 모든 스테이징 테스트는 `tmp` 디렉터리 등 격리된 환경에서 수행합니다.
- **Git 상태 전이:** 각 테스트 케이스 시작 전 `git reset --hard` 등으로 상태를 초기화하여 테스트 간 독립성을 보장합니다.

## 5. 검증 결과 요약

- **모든 항목 통과 시:** Phase Q 완료 및 Phase R (Git commit Agent) 구현 가능
- **실패 항목 존재 시:** argv 배열 전달 방식, `--` 구분자 사용 여부, 파일 경로 인코딩, 로거 연결 상태를 우선 점검합니다.
