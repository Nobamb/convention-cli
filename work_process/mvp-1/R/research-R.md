# Phase R Git Commit Agent Research

## 1. 개요

Phase R은 AI가 생성하고 사용자가 확인한 커밋 메시지를 사용하여 실제 Git 커밋을 수행하는 `commit(message)` 구현 기준을 정리하는 단계입니다. Git 스테이징(Phase Q) 이후 최종적으로 변경사항을 저장소에 기록합니다.

## 2. 작업 목표

- `src/core/git.js`에 `commit(message): void` 구현
- `git commit -m <message>` 명령 실행
- 보안을 위한 `execFileSync` 또는 `spawnSync` 기반의 argv 배열 방식 사용
- 한국어 및 이모지 처리를 위한 UTF-8 인코딩 보장
- 커밋할 내용이 없거나 메시지가 비어 있는 예외 상황 처리

## 3. Git 명령 기준

커밋 명령은 아래 의미로 실행합니다.

```bash
git commit -m "<message>"
```

Node.js에서는 메시지를 쉘 해석 없이 안전하게 전달하기 위해 argv 배열 형식을 사용합니다.

```javascript
execFileSync("git", ["commit", "-m", message], {
  encoding: "utf8"
});
```

## 4. 공백 및 한글 파일명 처리

`git commit -m` 명령은 파일 경로를 직접 인자로 받지 않으나, 커밋 메시지 내부에 포함된 공백, 줄바꿈, 특수문자, 한글이 쉘에 의해 잘못 해석되지 않도록 처리해야 합니다.

- **금지 방식:** ``execSync(`git commit -m "${message}"`)`` (Shell Injection 위험)
- **허용 방식:** `execFileSync("git", ["commit", "-m", message], ...)` (argv 배열 전달로 안전성 확보)

이 방식은 메시지 내용이 무엇이든 Git에 단일 인자로 정확히 전달됨을 보장합니다.

## 5. 에러 처리 기준

- **메시지 부재:** `message`가 없거나 빈 문자열인 경우 `Error`를 던지거나 실행을 중단합니다.
- **커밋할 내용 없음:** 스테이징된 변경사항이 없을 때 Git은 에러를 반환합니다. 이 에러 메시지를 캡처하여 사용자에게 알리거나 적절히 전파합니다.
- **Git 명령 실패:** 저장소 권한 문제, 락(Lock) 파일 존재 등 Git 명령 자체가 실패하는 경우 에러를 숨기지 않고 호출자에게 전달합니다.

## 6. 보안 및 로그 기준

- **Shell Injection 방지:** 반드시 argv 배열 방식을 사용하여 커밋 메시지 내의 특수문자가 실행 코드로 해석되는 것을 차단합니다.
- **로그 출력:** 커밋 성공 시 생성된 커밋 해시나 메시지 요약은 출력할 수 있으나, 메시지에 포함될 수 있는 민감 정보 노출에 유의합니다.
- **테스트 격리:** 실제 사용자 저장소에 영향을 주지 않도록 반드시 독립된 임시 테스트 저장소에서 검증합니다.

## 7. 테스트 관점

- 기본 영문 커밋 메시지 작동 여부
- 한글 및 다국어 커밋 메시지 깨짐 현상 확인
- 이모지(Emoji) 포함 메시지 처리
- 여러 줄(Multiline) 커밋 메시지 처리
- 스테이징된 파일이 없을 때의 동작
- 빈 메시지 입력 시의 동작
- 커밋 완료 후 `git log`를 통한 실제 기록 일치 여부

## 8. 다음 단계

Phase R이 완료되면 커밋된 내용을 원격 저장소에 반영하는 Phase S(Push Agent) 또는 커밋을 취소하는 Phase T(Reset Agent) 등의 부가 기능으로 연결됩니다.
