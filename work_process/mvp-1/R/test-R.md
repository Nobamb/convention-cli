# Phase R Git Commit Agent Test

`research-R.md`의 구현 기준을 바탕으로, `commit(message)`가 1차 MVP 요구사항을 충족하는지 확인하기 위한 테스트 항목입니다.

## 1. 정적 검증 항목

| ID | 테스트 항목 | 검증 방법 | 예상 결과 |
| :--- | :--- | :--- | :--- |
| V-1 | `commit(message)` export 확인 | `src/core/git.js` 코드 확인 | 다른 모듈에서 `commit` 함수를 import할 수 있다. |
| V-2 | Git 명령 실행 방식 확인 | `execFileSync` 또는 `spawnSync` 검색 | `execSync`를 사용하지 않고 배열 인자 방식을 사용한다. |
| V-3 | `-m` 옵션 구조 확인 | Git args 확인 | `["commit", "-m", message]` 구조로 인자를 전달한다. |
| V-4 | UTF-8 설정 확인 | child process 옵션 확인 | `encoding: "utf8"`을 지정하여 한글/이모지를 지원한다. |
| V-5 | 입력 유효성 검사 확인 | 함수 구현 확인 | 메시지가 비어있거나 타입이 맞지 않으면 에러를 던진다. |

## 2. 기능 테스트 항목

### T-1: 기본 커밋 성공
- **준비:** 테스트 저장소에서 파일을 수정하고 `git add`를 수행한다.
- **실행:** `commit("feat: add normal feature")`를 호출한다.
- **예상 결과:** 에러 없이 종료되며, `git log -1` 실행 시 해당 메시지가 확인된다.

### T-2: 한글 커밋 메시지
- **준비:** 파일을 수정하고 `git add`를 수행한다.
- **실행:** `commit("기능: 한글 메시지 테스트")`를 호출한다.
- **예상 결과:** 한글이 깨지지 않고 정상적으로 커밋 로그에 기록된다.

### T-3: 이모지 포함 메시지
- **준비:** 파일을 수정하고 `git add`를 수행한다.
- **실행:** `commit("feat: add emoji 🚀✨")`를 호출한다.
- **예상 결과:** 이모지가 정상적으로 저장되고 표시된다.

### T-4: 여러 줄(Multiline) 메시지
- **준비:** 파일을 수정하고 `git add`를 수행한다.
- **실행:** `commit("header\n\n- body line 1\n- body line 2")`를 호출한다.
- **예상 결과:** 개행 문자가 보존되어 커밋 본문에 기록된다.

### T-5: 커밋할 내용이 없는 경우
- **준비:** 스테이징된 변경사항이 없는 상태를 만든다.
- **실행:** `commit("should fail")`를 호출한다.
- **예상 결과:** Git 에러(nothing to commit)가 발생하며 함수가 에러를 전파한다.

### T-6: 빈 메시지 전달
- **준비:** 파일을 수정하고 `git add`를 수행한다.
- **실행:** `commit("")` 또는 `commit(null)`을 호출한다.
- **예상 결과:** 함수 자체에서 유효성 검사 에러를 던진다.

### T-7: 특수문자 포함 메시지
- **준비:** 파일을 수정하고 `git add`를 수행한다.
- **실행:** `commit("feat: test quotes ' \" $ ` ;")`를 호출한다.
- **예상 결과:** 쉘에 의해 해석되지 않고 문자열 그대로 커밋 메시지에 기록된다.

## 3. 테스트 절차

1. 현재 디렉터리를 백업하고 임시 디렉터리(`tmp/convention-commit-test`)로 이동한다.
2. `git init`을 수행하고 `user.name`, `user.email`을 로컬로 설정한다.
3. 초기 파일 생성 및 첫 번째 커밋을 완료하여 깨끗한 상태를 만든다.
4. 각 기능 테스트 케이스(T-1 ~ T-7)를 순차적으로 수행한다.
5. 각 호출 후 `git log -1 --pretty=%B` 명령을 통해 실제 기록된 메시지를 검증한다.
6. 예외 케이스(T-5, T-6)에서는 에러가 정상적으로 발생하는지 `assert.throws` 등으로 확인한다.
7. 테스트 완료 후 임시 디렉터리를 삭제하고 원래 디렉터리로 복구한다.

## 4. 테스트 환경 주의사항

- **운영 저장소 주의:** 절대 실제 개발 중인 `convention-cli` 저장소에서 테스트하지 않는다.
- **Git 설정:** 테스트 저장소의 `core.quotepath` 설정 등에 영향을 받지 않는지 확인한다.
- **환경 변수:** `GIT_AUTHOR_NAME` 등이 설정된 환경에서도 일관되게 동작하는지 확인한다.
- **이모지 지원:** 테스트를 실행하는 터미널/OS 환경이 UTF-8 및 이모지 출력을 지원하는지 확인한다.

## 5. 검증 결과 요약

- **모든 항목 통과 시:** Phase R 완료 및 Phase W 전체 커밋 플로우 통합 가능
- **실패 항목 존재 시:** `execFileSync` 인자 전달 방식, UTF-8 인코딩 설정, 빈 메시지 예외 처리 로직을 우선 점검한다.
